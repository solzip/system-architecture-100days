## Day 3 — 결합도(Coupling)와 응집도(Cohesion)

### 오늘 배운 것

응집도(Cohesion)는 모듈 내부 요소들이 하나의 목적에 얼마나 집중하는지를 나타내는 척도다.
7단계(우연적 → 기능적)로 분류되며, 기능적 응집도가 가장 이상적이다.

결합도(Coupling)는 모듈 간 의존 정도를 나타내는 척도다.
6단계(내용 → 메시지)로 분류되며, 메시지/데이터 결합이 가장 느슨하다.

Day 1의 SOLID 원칙과 Day 2의 관심사 분리는 결국 "높은 응집도 + 낮은 결합도"를 달성하기 위한 구체적 기법이었다.

| SOLID 원칙 | 영향 |
|-----------|------|
| SRP | 응집도를 높인다 — 변경 이유 1개 = 기능적 응집도 |
| OCP | 결합도를 낮춘다 — 확장 시 기존 코드 수정 없음 = 제어 결합 제거 |
| LSP | 결합도를 유지한다 — 자식이 부모를 대체 가능하면 호출자 결합이 깨지지 않음 |
| ISP | 결합도를 낮춘다 — 불필요한 메서드 의존 제거 = 스탬프 결합 방지 |
| DIP | 결합도를 낮춘다 — 추상화에 의존 = 내용/공통 결합을 데이터 결합으로 전환 |

### 왜 중요한가

마이크로서비스 경계를 정하는 기준이 "높은 응집도 단위로 쪼개고, 낮은 결합도로 연결"하는 것이다.
이 감각이 없으면 서비스를 잘못된 경계에서 잘라서 분산 모놀리스가 되어버린다.
클래스 수준의 원칙이 레이어(Controller/Service/Repository) → 서비스(마이크로서비스) → 시스템(Kafka 이벤트) 수준으로 동일하게 확장된다.

```
클래스 수준   →  SRP + DIP
레이어 수준   →  관심사 분리 (Day 2)
서비스 수준   →  마이크로서비스 경계 (Phase 2)
시스템 수준   →  이벤트 드리븐 / Kafka (Phase 2)
```

### 실습 내용

UserManager 리팩터링 (Before.java → After.java)

**Before 문제점:**
- 응집도: 우연적~절차적 (검증, 해싱, 저장, 이메일, 로깅이 한 메서드에)
- 결합도: 내용 결합(해싱 세부구현) + 공통 결합(전역 Map)
- SRP 위반: 변경 이유 5가지
- DIP 위반: 모든 것이 구체 구현에 직접 의존

**After 개선:**
- 응집도: 기능적 (각 클래스가 하나의 명확한 기능)
- 결합도: 데이터 결합 (인터페이스를 통한 파라미터 전달)
- SRP 준수: 각 클래스당 변경 이유 1개
- DIP 준수: 모든 의존이 인터페이스를 통과
